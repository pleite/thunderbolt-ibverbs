Date: 2026-05-21

Hosts:

- client: `strix-1`, kernel `7.1.0-rc1`
- server: `strix-2`, kernel `7.1.0-rc1`

## CPU-posted perftest baseline

Command family:

```sh
ib_*_bw -d usb4_rdma0 -x 1 -F --report_gbits -s 65536 -D 10 -q <qps> -t 64
```

For `ib_send_bw`, `--rx-depth 512` was also used.

| case | avg Gb/s | client process CPU | client host CPU | server process CPU | server host CPU |
| --- | ---: | ---: | ---: | ---: | ---: |
| `ib_write_bw`, q=1 | 10.74 | 90.5% | 3.9% | 0.1% | 1.1% |
| `ib_write_bw`, q=2 | 21.60 | 90.4% | 5.3% | 0.0% | 2.4% |
| `ib_send_bw`, q=1 | 10.98 | 99.4% | 4.3% | 90.7% | 4.1% |
| `ib_send_bw`, q=2 | 21.58 | 99.7% | 5.9% | 90.9% | 5.7% |

Interpretation:

- CPU-posted q=2 reaches about 21.6 Gb/s, but it does so with roughly one
  userspace CPU saturated on the active posting side.
- On a nominal 40 Gb/s TB4 link this still leaves substantial link headroom, so
  CPU posting may be the bottleneck. GDA could therefore be a throughput rescue
  as well as a latency/launch-overhead and CPU-offload path if it moves the
  limit to the kernel TX worker or the link.
- SEND burns one userspace CPU on both peers because the receive side is actively involved.
- Only q=1/q=2 were measured in this slice. Retest q=16/q=64/q=128 once both
  Strix hosts are free; if deeper queues do not improve bandwidth, the
  CPU-offload case for GDA gets stronger.

## HIP GPU/CPU coherency

Probe: `tbv-hip-gda-probes/bin/hip_coherence_probe`.

The probe does two descriptor-like handshakes:

- GPU writes payload, then publishes `gpu_tail` with `__HIP_MEMORY_SCOPE_SYSTEM` release semantics; CPU observes with acquire.
- CPU writes payload, then publishes `cpu_tail` with release semantics; GPU observes with `__HIP_MEMORY_SCOPE_SYSTEM` acquire semantics.

Results:

| allocation | iterations | GPU -> CPU | CPU -> GPU |
| --- | ---: | --- | --- |
| `managed` | 10000 | OK | OK |
| `host` | 10000 | OK | OK |
| `host-coherent` | 10000 | OK | OK |
| `host-uncached` | 10000 | OK | OK |
| `host-noncoherent` | 100 | FAIL (`gpu_error=1`) | FAIL (`gpu_error=2`) |

ABI consequence:

- Queue memory for v1 should require host-visible coherent or uncached HIP memory.
- `hipHostMallocNonCoherent` must be rejected for SQ/CQ/doorbell pages.
- The ABI comments should require GPU system-scope release/acquire for queue
  head/tail words; device/agent-scope atomics are not sufficient.
- Validated queue-memory allocators are `hipMallocManaged`, plain mapped
  `hipHostMalloc`, `hipHostMallocCoherent`, and `hipHostMallocUncached`.
- The current probes use 64-byte-isolated handshake words. They do not
  separately test HSA fine-grained pool allocation or any alignment requirement
  beyond the ABI's 64-byte SQ/CQ stride and 4 KiB doorbell-page alignment.

The same matrix was rechecked on `strix-2` after loading the DV-capable module
with `bind_services=0`; results matched the table above.

| CPU | CPU | `0x00000002` fine-grained | 4096 | 4096 | 4096 | OK | OK |

Interpretation:

- HSA CPU-location fine-grained and extended-scope fine-grained pools are
  usable queue-memory candidates on Strix Halo: returned pointers are 4 KiB
  aligned, satisfy the dedicated doorbell-page alignment, and can be registered
  with the current normal `ibv_reg_mr` path.
- The HSA GPU-location extended-scope pool is aligned and CPU/GPU-accessible,
  but normal `ibv_reg_mr` still fails with `EFAULT`. This matches the

  should request CPU/GPU agent access only. Passing the AIE agent to
  `hsa_amd_agents_allow_access()` is not needed for the GPU queue contract.


This proves:

- the driver-private uverbs method is discoverable via `driver_def`
- a standalone userspace process can issue `RDMA_VERBS_IOCTL` directly against
  `usb4_rdma0`
- the v1 caps shape matches the draft WQE/CQE/doorbell ABI
- caps are reported as ceilings plus recommended defaults, not fixed
  allocations
- the doorbell ABI distinguishes the 128-byte record from the dedicated 4 KiB
  page-sized mapping
- the doorbell record splits GPU-written producer fields from kernel-written
  consumer fields across two 64-byte cachelines
- the producer and consumer lines both carry explicit generation fields, and
  WQEs carry `generation` at offset 44, so KICK can reject stale WQEs and stale
  producer doorbell updates
- a CPU userspace process can create normal PD/CQ/QP objects, register
  page-aligned host buffers, attach them as the DV queue surface, and detach
  them cleanly

KICK design note:

- KICK should be implemented as a drain trigger: consume all valid WQEs between
  SQ head and the observed SQ tail. If SQ head equals SQ tail, it should return
  without expensive locking. The later poll-worker path should call the same
  drain helper and only change the trigger source.
- v1 does not define a DV receive queue. The receive queue remains
  kernel-owned and can be posted through standard verbs while the peer QP has
  an active DV queue. SEND/SEND_IMM and RDMA_WRITE_WITH_IMM consume peer
  receive WQEs and complete on the peer's normal verbs receive CQ. RDMA_WRITE
  does not consume a peer receive WQE.

## Phase 1 KICK smoke

KICK now pins userspace-provided SQ/CQ/doorbell memory at `CREATE_QUEUE`, then
uses those pinned pages for kernel-side SQ reads and CQ/doorbell writes. This
keeps the v1 ABI in the rocm-xio-friendly "userspace allocates queue memory"
shape while allowing asynchronous completions to be written outside the KICK
ioctl context.

Policy choices baked into the ABI and implementation:

- DV CQEs are produced in SQ WQE order. Internal native completions can arrive
  out of order; the kernel buffers them until prior WQEs are complete or skipped
  as unsignaled successes.
- A WQE generation mismatch produces an observable CQE with status
  `USB4_RDMA_DV_CQE_STALE_GEN` and advances SQ head past that WQE.
- DV CQ overflow moves the QP to ERR, updates `consumer.qp_state` when possible,
  and stops posting DV CQEs. No sentinel CQE is guaranteed when the CQ is full.

Live no-peer smoke on `strix-2` with `bind_services=0`:

```text
create_queue qp_num=2304 generation=1 sq_entries=256 cq_entries=512
kick_nop wr_id=0x4b49434b4e4f5031 status=0 opcode=0 sq_head=1 cq_tail=1 qp_state=0
kick_stale_gen wr_id=0x4b49434b5354414c status=6 opcode=0 sq_head=2 cq_tail=2 qp_state=0
destroy_queue ok
```

This validates host-side KICK plumbing without a peer: descriptor parsing,
generation checks, strict CQE publication, SQ/CQ doorbell updates, and pinned
queue-memory writes. This does not exercise the real transport; the SEND
payload smoke below covers that with both Strix hosts.

## Phase 2 kernel-poll SEND smoke

Added a device-level DV poll worker:

- one worker per `tbv_state`, covering the registered USB4 verbs device(s)
- fixed periodic cadence, default `dv_poll_interval_us=10`
- scan budget, default `dv_poll_budget=64` WQEs per worker pass
- round-robin QP list so a busy QP is moved to the tail before draining
- KICK and polling both call the same budgeted SQ drain helper

Teardown rule:

- `CREATE_QUEUE` attaches the QP to the poll list only after queue setup and
  response copyout succeeds.
- `DESTROY_QUEUE` and QP destroy take `dv_mutex`, remove the QP from the poll
  list, bump the generation, then clear/unpin queue resources.
- The poll worker must hold `dv_mutex` while touching SQ/CQ/doorbell memory.
  This mutex is the teardown acknowledgement: destroy waits for an in-flight
  poller before unpinning, and a poller that had only taken a QP ref before
  detach will acquire `dv_mutex` later, see the queue inactive, and touch no
  queue memory.

The SEND probe now supports `--no-kick`, which writes the WQE and publishes
`producer.sq_tail` but never issues `USB4_RDMA_DV_METHOD_KICK`.

## Phase 0.6 DV producer coherency probe

Extended `hip_coherence_probe` with `dv-producer` mode. This uses the real
`usb4_rdma_dv_wqe` and split `usb4_rdma_dv_doorbell` structs:

1. GPU writes every WQE field, including generation and reserved pattern words.
2. GPU publishes `producer.sq_tail` with a system-scope release atomic.
3. CPU waits with an acquire load of `producer.sq_tail`.
4. CPU validates the WQE bytes, then writes `consumer.sq_head` as the
   acknowledgement.
5. GPU waits on that acknowledgement before reusing the WQE slot.

Live `strix-2` results, 100,000 iterations:

```text
kind=host-coherent direction=dv-producer status=OK mismatches=0
kind=managed       direction=dv-producer status=OK mismatches=0
kind=host-uncached direction=dv-producer status=OK mismatches=0
kind=host          direction=dv-producer status=OK mismatches=0
```

This validates the exact Phase 3 producer-ordering pattern for the allowed HIP
queue-memory allocators. It does not remove the separate dmabuf/MR blocker for
GPU-location payload buffers.

## Phase 0.7 kernel-write to GPU direct-load probe

Added `hip_dv_kernel_cqe_probe`. This checks the reverse control-memory
direction needed by rocSHMEM pSync/signal words. The probe allocates DV
SQ/CQ/doorbell memory with a selected HIP allocator, attaches it to a normal
QP, repeatedly asks the kernel to complete NOP WQEs, and has a HIP kernel poll
the kernel-written `doorbell.consumer.cq_tail` with a direct volatile load. No
USB4 atomic-fetch is used for the observed word.

Live `strix-2` results:

```text
kind=host-coherent iterations=1000      status=OK avg_us=2.152   gpu_spins=4566
kind=host          iterations=1000      status=OK avg_us=2.146   gpu_spins=4471
kind=managed       iterations=1000      status=OK avg_us=2.354   gpu_spins=4912
kind=host-uncached iterations=1000      status=OK avg_us=2.102   gpu_spins=4328
kind=host-noncoherent iterations=1000   status=OK avg_us=125.498 gpu_spins=1263474

kind=host-coherent iterations=100000    status=OK avg_us=2.109   gpu_spins=449845
kind=host-noncoherent iterations=10000  status=OK avg_us=126.982 gpu_spins=12799595
```

Conclusion:

- host-coherent, ordinary mapped host, managed, and host-uncached memory are
  viable for kernel-written/GPU-polled control words on Strix Halo
- host-noncoherent is functionally visible in this probe but far too expensive
  for control-plane polling; keep it invalid for queue/sync memory
- a split rocSHMEM layout is justified: keep payload buffers in GPU/dmabuf
  memory, but allocate pSync/signal/control words from host-coherent memory so
  GPU kernels can use direct loads instead of USB4 atomic-fetch for local waits

## Phase 3 policy decisions

Destroy semantics for v1:

- `DESTROY_QUEUE` is teardown, not drain.
- In-flight DV WQEs are discarded without FLUSH_ERR CQEs.
- Consumers that need standard completion accounting must stop producing, wait
  for SQ head and CQ tail to catch up, then destroy.

Payload memory for Phase 3:

- Phase 3 will use host-visible payload buffers first: `hipMallocManaged` or
  coherent/mapped host memory for payloads, and the same validated coherent
  allocator family for SQ/CQ/doorbell memory.
- dmabuf-backed GPU-location payload buffers are Phase 3.5. The descriptor ABI,
  queue allocation, and rocm-xio provider mechanics are independent of whether
  the payload lkey/rkey came from ordinary `ibv_reg_mr()` or future
  `ibv_reg_dmabuf_mr()`.

rocm-xio provider shape:

- Add `Provider::USB4_RDMA`, provider strings `usb4`, `usb4_rdma`, and a
  `GDA_USB4_RDMA` CMake option.
- Keep USB4 inside the existing `rdma-ep` backend model. Host orchestration
  belongs in a new `src/endpoints/rdma-ep/usb4/usb4-backend.cpp`, mirroring
  the ERNIC split: open verbs device, create ordinary QP/CQ/PD, transition QP,
  allocate SQ/CQ/doorbell queue memory, call the USB4 private uverbs
  `CREATE_QUEUE`, then initialize GPU-visible `QueuePair` state.
- Add `usb4-provider.hpp` for GPU-visible queue state: WQE/CQE/doorbell
  pointers, queue depths, generation, local producer indices, and lkey/rkey.
- Add `usb4-queue-pair.hip` implementing provider `Ops`:
  - `put_nbi` / `put_nbi_imm` as `RDMA_WRITE` / `RDMA_WRITE_IMM` WQEs.
  - `quiet` / `quiet_single` as DV CQ polling plus `producer.cq_head` update.
  - no READ or atomics in the first provider slice; reject or leave unsupported
    rather than pretending USB4 v1 implements them.
- Use host-visible coherent queue memory only for Phase 3. If rocm-xio requests
  device queue memory for this provider, reject or force host-coherent mode.
- Use ordinary `ibv_reg_mr()` for Phase 3 payload buffers. Managed/coherent host
  payloads are acceptable; GPU-location/dmabuf payloads wait for Phase 3.5.
- Do not require a separate provider DV shared library. The USB4 ABI is already
  a provider-private uverbs interface, so the backend can issue the same raw
  ioctls as the probes unless/until those helpers move into rdma-core.

## Device-memory registration / dmabuf gap

The HIP registration probe shows that `hipMalloc` device memory fails the
current `ibv_reg_mr` path with `errno=14 Bad address`. On Strix Halo this is a
driver-pinning/API gap rather than a hard hardware limitation: device memory is
UMA-backed DRAM, but the normal user-VA pinning path does not accept the HIP
device mapping.

Local code inspection:

- The usb4 kernel driver currently implements only `.reg_user_mr`, and
  `tbv_reg_user_mr()` calls `ib_umem_get()` on a userspace VA.
- The driver does not implement `.reg_user_mr_dmabuf`.
- Linux RDMA core has `ib_umem_dmabuf_get*()` helpers and the ib_device op
  `.reg_user_mr_dmabuf`.
- libibverbs already exposes `ibv_reg_dmabuf_mr()`.
- The usb4 rdma-core provider currently wires `.reg_mr`, not `.reg_dmabuf_mr`.
- The KFD UAPI has dmabuf export/import structures
  (`kfd_ioctl_export_dmabuf_args`, `kfd_ioctl_import_dmabuf_args`,
  `kfd_ioctl_get_dmabuf_info_args`).

Likely work:

- Status as of Phase 2: parked. No dmabuf code was added in the poll-worker
  slice, but this remains the long pole before Phase 3 can use GPU-location
  payload buffers instead of host-visible or managed memory.
- Add a userspace probe that can obtain a dmabuf fd for a ROCm allocation, then
  call `ibv_reg_dmabuf_mr()` against `usb4_rdma0`.
- Add `usb4_rdma` provider `.reg_dmabuf_mr` support using
  `ibv_cmd_reg_dmabuf_mr()`.
- Add kernel `.reg_user_mr_dmabuf` support using `ib_umem_dmabuf_get_pinned*()`
  and map/release the dmabuf-backed umem correctly.
- Audit the data path before claiming discrete VRAM support: today the soft-RNIC
  copies through `ib_umem_copy_from()` and `kmap_local_page()` over `sg_page()`.
  That is plausible for Strix UMA system memory, but a discrete 6700 XT VRAM
  dmabuf may need a different CPU mapping or copy path.

Reload caveat:

- A full `strix-1` reload with service binding enabled selected a generic
  `7.1.0-rc1` dev tree, compiled the legacy source-blind Thunderbolt callback
  path, and wedged during module init in `tbv_peer_get_or_create()` before
  verbs registration. That is not on the DV ioctl path. Future live tests
  should pin `KDIR` to the booted `linux-usb4` dev tree or use
  `bind_services=0` when only testing uverbs ABI plumbing.
- Added two guardrails after reproducing the wedge:
  - `thunderbolt-ibverbs-build` now prefers source-aware kernel dev trees and
    refuses ambiguous source-aware matches unless `KDIR` is explicit.
  - `thunderbolt-ibverbs-reload` and the driver refuse source-blind native
    negotiation/tunnel mode by default. The explicit override is
    `TBV_ALLOW_SOURCE_BLIND_NATIVE=1` / `allow_source_blind_native=1`.
- `strix-1` recovered only after direct kexec/power-cycle style recovery. The
  staged kexec image loaded, but normal `kexec.target` stuck because the
  D-state `insmod` remained in `session-16.scope` and systemd waited for the
  scope to stop. Do not rely on clean systemd kexec once this module-init wedge
  has happened.

## Phase 3 WRITE_WITH_IMM smoke

The first WRITE_IMM attempt exposed an ABI/policy bug: `ibv_post_recv()` on the
target QP returned `EBUSY` after `CREATE_QUEUE` because the driver treated an
active DV queue as blocking every standard verbs post. For v1 that was too
strict. DV owns the GPU-produced SQ and DV CQ surface; the receive queue is
still kernel-owned, so normal receive WQEs must remain postable for
SEND/SEND_IMM and WRITE_WITH_IMM.

Driver fix:

- keep standard send posting blocked while `dv_queue_active`
- allow `tbv_post_recv()` while `dv_queue_active`
- document in the ABI header that RQ is kernel-owned in v1

rocm-xio changes:

- added `--write-imm` for the two-node `rdma-ep` WRITE correctness test
- USB4 QPs now create a small RQ (`max_recv_wr=16`) for immediate completions

- client launches a HIP `RDMA_WRITE_WITH_IMM` WQE
- server polls its normal verbs CQ for `IBV_WC_RECV_RDMA_WITH_IMM`, validates
  immediate data `0x58494f01`, then verifies payload bytes

## Phase 3 in-process multi-QP rocm-xio throughput

Added an in-process two-node `--throughput --num-queues N` path to rocm-xio.
The client creates N USB4 GDA QPs in one process, passes all QP/source/dest

## Phase 3.5 dmabuf payload MR smoke

Added dmabuf MR support to the USB4 userspace provider and kernel driver, then
allowed rocm-xio's two-node throughput path to allocate/register DATA_DEVICE
payload buffers with `ibv_reg_dmabuf_mr`.

Key fixes:

- Provider implements `.reg_dmabuf_mr`; without this, `ibv_reg_dmabuf_mr`
  returned `EOPNOTSUPP` before entering the kernel.
- Kernel implements `.reg_user_mr_dmabuf`.
- dmabuf MRs use `virt_addr` as the WQE IOVA and `start` as the exported
  dmabuf offset. Local-send SGE validation now translates IOVA to umem offset.
- Imported dmabuf sg tables can have `nents > 0` and `orig_nents == 0`; the
  driver now walks `sgt->nents` for user-memory copy/page helpers.

Registration probe:

```text
strix-2: hip_reg_mr_probe device 4096 gpu-touch dmabuf -> reg_mr=OK lkey=0x2 rkey=0x2
```

## Device-source DMA visibility probe

Question: can the Thunderbolt NHI DMA engine read GPU-written device payloads
directly, without a `hipDeviceSynchronize()` or an intermediate host-coherent
copy?

Probe:

```sh
tbv-hip-gda-probes/bin/hip_rdma_write_visibility_probe \
  --device usb4_rdma0 --gid-index 1 --role send|recv \
  --source-kind <kind> --source-fill <fill> --source-reg dmabuf \
  --size <bytes> --count <n>
```

The sender fills a source buffer from a HIP kernel, registers it with
`ibv_reg_dmabuf_mr()`, and sends it over the normal USB4 zcopy data path. The
receiver checks every byte and a per-iteration signal word.

Results on `strix-2 -> strix-1`:

| source allocation | fill / ordering | 4 KiB x 10 | 64 KiB x 10 | sustained result |
| --- | --- | --- | --- | --- |
| `hipMalloc` device | GPU fill only | FAIL, stale/zero payload | not retested | requires HIP runtime sync |
| `hipMalloc` device | GPU fill + `hipDeviceSynchronize()` | OK | not retested | OK in short tests |
| HIP VMM device pinned | GPU fill + HDP read | FAIL | not retested | no better than `hipMalloc` |
| HIP VMM device uncached | GPU fill + HDP read | FAIL | not retested | no better than `hipMalloc` |
| HIP VMM host pinned | GPU fill + HDP read | FAIL | not retested | no usable external-DMA visibility |
| HIP VMM host uncached | GPU fill + HDP read | FAIL | not retested | no usable external-DMA visibility |
| HSA GPU-location extended-scope fine-grained pool | GPU fill only | OK | FAIL around WR 41/100 | 4 KiB x 1000 failed near WR 985 |
| `hipExtMallocWithFlags(..., hipDeviceMallocFinegrained)` | GPU fill only | FAIL | not retested | not a replacement for sync |
| `hipExtMallocWithFlags(..., hipDeviceMallocFinegrained)` | GPU fill + `hipDeviceSynchronize()` | OK | not retested | OK in short tests |
| `hipExtMallocWithFlags(..., hipDeviceMallocUncached)` | GPU fill only | OK | OK | 4 KiB x 100 OK; 64 KiB x 100 failed around WR 81 |

Key observations:

- The Thunderbolt driver is already doing real zcopy DMA for large registered
  payloads: the path maps MR pages with `dma_map_page(..., DMA_TO_DEVICE)` and
  `data_wr_zcopy` increments during these runs.
- The unreliable part is not dmabuf registration itself. It is GPU-store
  visibility and ordering before an external DMA engine reads those pages.
- Ordinary `hipMalloc`, HIP VMM pinned, HIP VMM uncached, and HIP
  fine-grained allocations do not make GPU kernel stores reliably visible to
  NHI DMA without a HIP runtime synchronization point.
- The HSA GPU-location extended-scope pool proves that unsynchronized external
  DMA can work on Strix Halo, but the current path is slow and not robust under
  sustained zcopy pressure.
- `hipDeviceMallocUncached` is the best candidate found so far: no-sync GPU
  fills passed 32 B, 4 KiB, and 64 KiB short runs, and a 4 KiB x 100 run. It
  still triggered a transport timeout in the 64 KiB x 100 run, so it should
  stay experimental until the timeout is understood.

Current policy:

- Keep host-coherent rocSHMEM scratch as the reliable default for application
  smoke tests.
- Add an opt-in uncached-device scratch variant for experiments that need the
  real device-source DMA path.
- Do not claim arbitrary application `hipMalloc` tensor memory can be used as a
  no-sync RDMA source. Today that requires either a HIP synchronization point,
  allocation control (`hipDeviceMallocUncached`), or further ROCm/driver
  support for a proper GPU-to-external-DMA visibility transition.

Follow-up packaging/application smoke:

- Added an opt-in RCCL/rocSHMEM source-heap path:
  `RCCL_ROCSHMEM_SOURCE_HEAP=1` allocates RCCL source scratch with
  `rocshmem_malloc()` instead of `rocshmem_malloc_host_coherent()`.
- Built `rocshmem-usb4-uncached-gfx1151`, where the rocSHMEM heap uses
  `hipExtMallocWithFlags(..., hipDeviceMallocUncached)`.
- Built `rccl-tests-usb4-uncached-gfx1151`; the wrapper sets
  `RCCL_ROCSHMEM_SOURCE_HEAP=1` by default.
- Two-node `all_reduce_perf` passed on `strix-1,strix-2` with
  `RCCL_ROCSHMEM_SOURCE_HEAP=1`; logs showed `source heap=...` on both ranks.
  This validates RCCL initialization and the allocator switch, but the
  collective itself used RCCL NET/IB.
- Two-node `alltoall_perf -b 65536 -e 65536 -n 1` passed with `#wrong=N/A`
